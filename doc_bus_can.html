Documentation technique – Banc d’essai BUS CAN / LIN

1. Comprendre l’architecture globale

1.1 Description générale du banc

Le banc Leybold “Training Panel Lighting” reproduit une partie du réseau électronique du domaine habitacle d’une Audi A4 (2007). Il intègre un réseau CAN, un réseau LIN, un système d’éclairage, une commande de tableau de bord (combiné) et une prise de diagnostic OBD pour les opérations de test/diagnostic. :contentReference[oaicite:5]{index=5}

Sur la maquette, les calculateurs principaux étudiés sont : J519, J393, J345, J527 et J285. La maquette est alimentée en 12/14 V (douille noire = masse, douille 30 = + permanent). :contentReference[oaicite:6]{index=6}

  • J519 : centrale électrique / réseau de bord (pilotage éclairage, interface CAN/LIN).
  • J527 : électronique de colonne (commandes conducteur, comodo… → messages CAN).
  • J285 : combiné d’instruments (voyants/indications selon les états CAN).
  • J393 : calculateur central de confort (fonctions confort, échanges sur CAN Confort).
  • J345 : identification de remorque (selon configuration/usage didactique).

1.2 Topologie globale du réseau automobile (CAN/LIN)

Un véhicule moderne n’utilise pas “un seul CAN” mais plusieurs sous-réseaux (souvent plusieurs CAN spécialisés) et des bus secondaires. Une architecture typique distingue notamment :

  • CAN Propulsion (groupe motopropulseur),
  • CAN Confort (habitacle : éclairage, verrouillage, commandes…),
  • CAN Diagnostic (accès diagnostic via la prise de diagnostic),
  • Bus LIN (bus maître/esclaves pour capteurs/organes simples),
  • et éventuellement MOST (infotainment) et autres sous-systèmes selon équipement.

Cette séparation permet d’adapter le débit/robustesse selon les fonctions et de limiter les interférences entre domaines (confort vs propulsion, etc.). :contentReference[oaicite:7]{index=7}

1.3 Topologie du réseau habitacle (Convenience Electronics)

Le domaine habitacle (“Convenience/Comfort”) regroupe les fonctions liées au confort et à l’interface conducteur (éclairage, commande colonne, combiné, confort…). Les calculateurs échangent des états sur le CAN Confort, et certains sous-systèmes sont raccordés via LIN (capteurs/commandes), typiquement avec un calculateur jouant le rôle de maître LIN.

Dans ce domaine, une topologie fréquente est une convergence “en étoile” vers un point de jonction (node). Un avantage est qu’une panne d’un module n’empêche pas forcément les autres de continuer à échanger. En cas de court-circuit à la masse/au plus/entre lignes, le CAN Confort peut passer en emergency running mode (mode dégradé, possible fonctionnement en “single-wire” selon transceiver). :contentReference[oaicite:9]{index=9}

Sur votre banc, les défauts réseau typiques sont explicitement simulables (court-circuit CAN High au +, CAN Low à la masse, ou CAN-H et CAN-L court-circuités). :contentReference[oaicite:10]{index=10}

1.4 Architecture spécifique du banc d’essai

La maquette Leybold utilisée en TP comporte un seul réseau CAN BUS configuré à 100 kbit/s. L’acquisition/analyse des trames se fait via une interface PEAK et le logiciel PCAN-View (bitrate 100 kbit/s). Le diagnostic “atelier” est réalisé via la prise OBD et un outil type VCDS. :contentReference[oaicite:11]{index=11}

Chaîne de test recommandée (vue “banc d’essai”) :

  • Alimentation 12/14 V → mise sous tension maquette,
  • Connexion OBD → (a) VCDS pour auto-scan / défauts, (b) interface CAN (PEAK) pour observer les trames,
  • Réseau CAN Confort (100 kbit/s) reliant J519, J393, J527, J285, J345,
  • Simulation de défauts via les “fault switches” (dont défauts de ligne CAN).

(À illustrer par un schéma : PC + PCAN-View ↔ OBD ↔ CAN Confort ↔ ECU, et PC + VCDS ↔ OBD.)

2. Documentation des outils

2.1 OBD CAN + USB Adapter

  • Fonction : Interface permettant de relier le PC au banc d’essai via la prise OBD afin d’accéder au bus CAN.
  • Schéma : PC → USB → Adaptateur OBD → Réseau CAN du banc.
  • Type de communication : USB côté PC, CAN côté banc.
  • Données manipulées : Trames CAN et requêtes de diagnostic.
  • Utilisation dans le banc : Connexion du logiciel de diagnostic pour scanner les calculateurs et lire ou effacer les défauts.

2.2 Logiciel VCDS

  • Fonction : Logiciel de diagnostic permettant d’identifier les calculateurs et de lire les codes défauts.
  • Schéma : PC (VCDS) → Adaptateur OBD/USB → Réseau CAN du banc.
  • Type de communication : Communication diagnostic via CAN (par interface OBD).
  • Données manipulées : Codes défauts (DTC), identification des calculateurs, états de fonctionnement.
  • Utilisation dans le banc : Scan du réseau, analyse des défauts et validation des scénarios de panne.

2.3 Sensor-CASSY 2

  • Fonction : Module de mesure permettant d’acquérir des signaux électriques à l’aide de capteurs.
  • Schéma : Capteur → Sensor-CASSY 2 → PC.
  • Type de communication : Acquisition de données via interface PC (USB).
  • Données manipulées : Signaux analogiques ou numériques (tensions, états).
  • Utilisation dans le banc : Observation et analyse des signaux lors des tests et des défauts simulés.

2.4 CAN-Databus-Multi-Adapter

  • Fonction : Interface permettant de se connecter directement au bus CAN pour observer les trames.
  • Schéma : PC → CAN multi-adapter → CAN-H / CAN-L du banc.
  • Type de communication : Communication CAN.
  • Données manipulées : Trames CAN (identifiants, données, erreurs).
  • Utilisation dans le banc : Analyse des échanges CAN et comparaison entre fonctionnement normal et dégradé.

2.5 LIN-Bus PC Interface USB

  • Fonction : Interface PC permettant de communiquer sur le réseau LIN.
  • Schéma : PC → USB → Interface LIN → Réseau LIN du banc.
  • Type de communication : USB côté PC, LIN côté banc.
  • Données manipulées : Trames LIN (messages maître-esclave).
  • Utilisation dans le banc : Observation et analyse de la communication LIN des sous-systèmes habitacle.
